# 爱心厨房（Love Kitchen）产品需求汇总 v1.1

**文档版本**：v1.1  
**用途**：产品共识、评审转发、研发对齐  
**核心设计理念**：极简交互，智能联动，家庭协作——能让系统代劳的，绝不让用户多点一次。

---

## 一、产品定位与目标

* **定位**：面向家庭的 **AI 厨房助手** 微信小程序，重点解决「今天吃什么、怎么做」以及 **食材浪费** 问题。
* **核心价值**：
    * **极简管理**：零摩擦录入与消耗库存，告别繁琐的手动记账。
    * **智能决策**：通过搜索与「今日推荐」降低决策成本；天气类推荐以**用户主动刷新**获取实时数据，避免静默定位与权限弹窗影响冷启动。
    * **高效执行**：全站统一提供 **结构化菜谱详情 + 图文烹饪步骤**。
    * **库存减损**：通过 **冰箱库存与临期管理** 提高食材周转，从根本上减少浪费。
    * **家庭协作**：基于微信生态的极轻量社交，一键转发实现家庭采购与烹饪的无缝对接。

---

## 二、信息架构（底部导航）

采用 **4 个 Tab**，并强化 Tab 2 与 Tab 3 的业务联动，形成功能闭环：

| Tab | 名称 | 核心职责与说明 |
| :--- | :--- | :--- |
| 1 | **首页** | 搜索（文字/语音/拍照）+ **零输入每日推荐** |
| 2 | **选菜** | **(核心主路径)** 默认展示“临期食材”，一键智能匹配消化库存的菜谱 |
| 3 | **冰箱** | **极简库存管理**：语音/拍照快捷入库，查看存放时间与临期信息 |
| 4 | **我的** | 常规配置：设置、历史、收藏、关于 |

---

## 三、核心功能与交互需求

### 3.1 冰箱页（Tab 3）：库存的“零摩擦”进出

* **语音快捷入库**：用户按住说话（如：“买了两个番茄和半斤排骨”），AI 自动提取结构化数据入库。
* **拍照快捷入库**：拍摄购物小票或食材袋，调用 AI 识别入库。
* **智能默认保质期**：摒弃手动输入保质期。系统根据食材类别提供默认天数（如绿叶菜默认 3 天），用户只需“一键确认”。
* **闭环自动出库**：在“菜谱详情”页点击“开始烹饪”后，系统弹窗提示“是否自动扣减消耗的食材库存？”，一键完成出库。

### 3.2 选菜页（Tab 2）：以“减损”为导向的智能联动

* **库菜联动默认态**：进入页面，顶部高亮区域直接展示 **冰箱内「即将过期」的食材**，作为首选匹配项。
* **操作步骤 1**：系统默认勾选 1-2 款临期食材（用户也可在下方补充勾选其他食材）。
* **操作步骤 2**：点击 **「智能匹配菜谱」**。
* **操作步骤 3**：进入 **匹配结果列表**，优先推荐能最大化消耗勾选食材的菜式。
* **操作步骤 4**：点击某道菜进入统一的 **菜谱详情**。

### 3.3 首页（Tab 1）：搜索 + 今日推荐

* **搜索入口**：文字搜索食材，结果进入匹配列表与「菜谱详情」。（语音/拍照按迭代接入。）
* **今日推荐**：默认使用**参考天气**生成推荐文案与列表，避免进入首页即请求定位。用户点击 **「刷新天气」** 后再请求实时天气（Open-Meteo 等），拒绝授权时仍可继续使用参考数据。
* **就餐人数**：默认取「我的」中设定（未设定则 2 人），用于 AI 用量描述。
* **展示形式**：以横向滑动等形式呈现推荐菜列表，与搜索区同页，减少重复入口。

### 3.4 我的（Tab 4）

* **核心配置**：设置（含 API Key 配置等）、历史记录、收藏夹、关于页面。

## 四、菜谱详情（全站统一）

* **统一结构**：各入口（搜索、推荐、匹配）共用同一套详情页，包含标题、标签、时间/难度、用料列表、**分步图文步骤**。
* **执行动作（开始烹饪）**：页面定位到步骤区，同时触发 **“库存联动扣减”** 逻辑。
* **轻社交动作（微信分享）**：详情页提供明显的**「转发给好友/微信群」**按钮。
* **分享卡片定制**：抓取菜谱主图为封面，自动生成诱导点击文案（如：“今晚吃这个👉【清蒸鲈鱼】”）。
* **家庭采购协作**：分享卡片附带所需食材简要列表，方便家人在微信聊天界面直接查看并前往菜市场采购。

## 五、非功能与降本策略 (架构生命线)

* **AI 缓存兜底机制**：建立标准菜谱的“本地/云端缓存库”。用户搜索常见菜时优先命中缓存库（秒开，零 Token 消耗）。仅当搜索长尾菜名或复杂剩余食材混搭时，才调用大模型 API 实时生成 JSON。
* **性能与体验保障**：列表与长图片务必做懒加载处理；弱网环境需有友好提示。
* **异常拦截处理**：网络失败、AI 超时、空冰箱、无匹配结果等场景，需提供 **明确文案** 与 **返回/重试** 路径。
* **统一 JSON 协议 Schema**：各场景返回的菜谱数据必须严格遵循预设 JSON 格式，前端需做强校验，彻底杜绝大模型返回纯文本导致页面白屏的风险。

## 六、v1 范围边界（刻意控制）

为保持产品 **聚焦、好用**，v1 MVP 版本明确边界如下：

* **拥抱轻社交**：全面支持基于微信原生能力的“一键转发”，利用家庭协作场景实现自然裂变。
* **拒绝重社区**：坚决不做应用内的评论区、打卡墙、用户互粉等需要重度运营的功能。
* **拒绝重运营**：不包含盲盒抽奖、周末专栏等复杂运营栏目。

---

## 七、技术架构对齐

* **前端框架**：Taro + React + TypeScript。
* **AI 引擎**：MiniMax（OpenAI 兼容，如 MiniMax-M2.7）；正式环境密钥经 **服务端中转** 持有，客户端不强制用户配置 Key。
* **交付形态**：以 **微信小程序** 为唯一主目标，环境配置、`package.json` 脚本与 README 规范需与微信开发者工具工作流完全一致。
